Story Generation Data Flow
🖱️

1 · Parent submits story request

POST /api/stories/generate with { childId, themePrompt? }. Frontend calls useSSE(storyId) immediately to open an EventSource connection for real-time updates.

🔒

2 · Auth + Quota validation

requireAuth middleware verifies the JWT cookie, hashes it, and checks the sessions table for a live, non-expired row. Then the route queries daily_quotas to enforce free-tier limits.

🛡️

3 · Pre-moderation of theme prompt

moderateText(themePrompt) runs 5 regex patterns for violence, adult content, drugs, profanity, and horror before the Claude API is ever called. If blocked, a MODERATION_BLOCKED error is thrown and the SSE client receives a recoverable error event.

📋

4 · DB record created → 202 returned immediately

A story row is inserted with status='generating'. The API responds 202 Accepted with the storyId. The client is already subscribed to SSE — it will receive events when generation completes. The actual generation runs in setImmediate(), keeping the HTTP request non-blocking.

🤖

5 · Claude API call (async background)

The system prompt encodes age-appropriate vocabulary rules, the child's name/interests, and the last 5 story themes to prevent repetition. The user prompt adds tonight's optional theme. claude-sonnet-4-5 responds with a structured JSON object: { title, themeSummary, pages[4] }.

🛡️

6 · Post-moderation on LLM output

All page text is joined and passed through the same moderateText() filter. If any pattern matches the AI output, getFallbackStory() replaces the generated content with a pre-written safe story. This double-gate ensures no harmful content reaches children even if the LLM misbehaves.

💾

7 · Persist to SQLite + emit SSE

Story row updated to status='ready' with token/latency/cost metadata. 4 rows inserted into story_pages. Theme summary saved to story_themes for future variation. Then emitSSE(storyId, { type: 'story_complete' }) fires — the client navigates to the story view.

📊

8 · Telemetry recorded throughout

trackEvent() writes to the events table at generation start, completion (with latency + cost), moderation blocks, and favorites. The app_sessions table tracks session duration and pages read. All surfaced via /api/metrics and the /dashboard HTML page.

Design Tradeoffs · Technical Insights · Risk Management
⚡
Async Story Generation (202 + SSE)
Architecture Tradeoff

Why: Claude can take 4–8s+ to generate a story. A synchronous response would time out or block the HTTP connection. By returning 202 immediately and pushing story_complete over SSE, the UI stays responsive and shows a live loading state.

Risk: In-memory SSE client Map is lost on server restart. Clients reconnecting after a restart won't receive the event — but the SSE handler falls back to polling: if the story is already status='ready', it emits immediately on reconnect.

🛡️
Double-Gate Content Moderation
Safety Design

Why: Regex pre-moderation stops bad prompts from reaching the Claude API (saves cost). Regex post-moderation catches the rare case where the LLM produces unsafe output despite the system prompt guardrails.

Risk: Regex is brittle — it can't understand context ("he shot a basket" would be blocked). Upgrading to a dedicated moderation API (e.g., Claude's own moderation or OpenAI Moderation) would reduce false positives while improving catch rate.

🗄️
SQLite with WAL Mode
Database Tradeoff

Why: Zero-config, embedded, no network overhead, fast for a single-server app. WAL mode allows concurrent readers while writes are happening — critical when telemetry writes happen alongside story fetches.

Scaling limit: SQLite doesn't support multiple write processes. At scale (multiple API instances), you'd need to migrate to Postgres. The synchronous better-sqlite3 writes also block the Node.js event loop — acceptable at low concurrency, but a bottleneck under load.

🔑
JWT + Token Hash Session Model
Auth Architecture

Why: JWTs are stateless and fast to verify. Storing the token hash in the DB (not the raw token) allows server-side revocation — you get the best of both: fast validation and the ability to log users out.

Note: httpOnly cookies protect against XSS. CORS is locked to :5173. The auth rate limit (10 req/min) prevents brute-force. No refresh token flow means sessions expire without renewal.

🎯
Theme Variation System
Product Insight

How: The last 5 story themes per child are fetched from story_themes and injected into the system prompt as "RECENT STORIES TO AVOID REPEATING." This costs only a tiny amount of extra tokens but significantly improves story novelty.

Future: Could be extended to track themes across a sliding window (30 days), or use semantic similarity to avoid thematically similar stories even if keyword-different.

💰
Cost Tracking + Freemium Model
Business Model

Every story records llm_prompt_tokens, llm_completion_tokens, and estimated_cost_usd. The metrics API surfaces P50/P95 latency, cost/story, and 7-day spend. Free tier caps at 3 stories/day + 10 saved (auto-prune oldest non-favorite).

Risk: Cost estimate is approximate (hardcoded rates). Token pricing changes should be updated in story-gen.ts:152. No billing integration yet — premium tier exists in DB schema but isn't enforced.

📊
Built-in Observability
Technical Insight

The telemetry layer tracks: story generation lifecycle, moderation blocks (with stage + reason), session length, pages read, and per-story cost. KPIs are computed via SQL queries in /api/metrics: stories/user/week, avg session length, safety pass rate.

Strength: No external analytics dependency. Everything is queryable from SQLite. The metrics dashboard auto-refreshes every 30s and requires only an authenticated session.

⚠️
Risk Register
Priority Issues to Address
HIGH No image generation: illustrationPrompt is stored but never rendered as an actual image. The UI shows text-only stories — a core product promise unimplemented.
HIGH Daily limit disabled: Freemium quota check is present in code but commented out ("disabled for testing") — all users have unlimited generations, burning real API budget.
HIGH SQLite single-file: No backup strategy. Data is on-disk only. A server crash without a backup = complete data loss.
MED In-memory SSE state: Server restart silently loses all pending story subscriptions. Clients must reconnect and poll.
MED Regex moderation false positives: Innocent words like "die" (as in coloring dye) or "shot" (basketball) will be blocked.
MED No HTTPS enforced: JWT cookies over HTTP in development. Production deployment must enforce HTTPS and set Secure cookie flag.
LOW No refresh token: Sessions expire without renewal. Long reading sessions could expire mid-story without a silent refresh mechanism.
LOW Cost rates hardcoded: Anthropic pricing changes would silently produce incorrect cost metrics until story-gen.ts:152 is updated.
